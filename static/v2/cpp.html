C++ Quickstart


Clone the repository and compile OpenCensus:

git clone https://github.com/census-instrumentation/opencensus-cpp.git
bazel build opencensus/...
bazel build examples/...

Running the HelloWorld Example:
./bazel-bin/opencensus/examples/helloworld/hello_world



Overview

Traces
A trace measures the amount of time it takes an application to complete a single operation. Traces usually extend across multiple nodes in a distributed system. For example, it can describe how long it takes for the application to process an incoming request from a user and return a response. This could include the time taken for the frontend to send requests and receive responses from backend systems. A trace consists of one or more spans, each of which describes the amount of time it takes to complete a suboperation. Traces are uniquely identified by a TraceId. All spans within a trace will share the same TraceId.



Spans
Spans measure the latency of suboperations within a trace. Multiple spans can be part of a "Trace", which represents an execution path (usually distributed) across multiple processes/nodes. For example, it can describe how long it takes for the application to perform a round-trip RPC call to another system when handling a request, or how long it takes to perform another task that is part of a larger operation. Spans are uniquely identified by a SpanId and a TraceId. Spans can inherit the TraceId from a parent span. If they are created without a parent, they will be marked as a root span. The initial span in a trace will always be a root span.

Spans may optionally contain a number of other elements that can be used for debugging and diagnostic purposes. This includes attributes, annotations, message events, links, and status.

Example span creation:

auto parent_span = opencensus::trace::Span::StartSpan("root");
auto child_span = opencensus::trace::Span::StartSpan("child", &parent_span);



Attributes
Attributes are additional information that is included in the span which can represent arbitrary data assigned by the user. They are key value pairs with the key being a string and the value being either a string, boolean, or integer.

Examples:

span.AddAttribute("my_attribute", "red");
span.AddAttribute("event_occurred", "true");
span.AddAttribute("bytes", "10");



Annotations
Annotations are a set of attributes that have been grouped together and are optionally accompanied by a string description.

span.AddAnnotation("Constructing greeting.", 
  {{"name", request->name()}, {"name length",
  request->name().size()}});



Message Events
Message events represent messages sent/received between spans. It records whether the message was sent/received and optionally the message size.



Links
Links represent a connection between two spans in different traces. The linked span can be either a parent or child.



Status
Status represents the current state of the span. It is represented by a canonical status code which maps onto a predefined set of error values and an optional string message.



Span Context
The span context represents the state information that must be propagated to child spans and between processes to maintain a trace. A span context contains a TraceId, SpanId, and a set of span options for a given span.
Example span context:
TraceId-SpanId-Options:
3563a30535bddcaeca5dd00bc84f29ba-90a9d220f553e7fc-01



Starting/Ending Spans
When a span starts it records a start time. When a span ends, the end time is recorded. Latency is calculated by the difference between the start and end times. Spans will only be exported to exporting services after they have ended. Attributes, annotations, message events, etc. can only be added to span that is active (i.e. has started but has not ended). Ending a span is usually the responsibility of the user.



Samplers
Tracing every span can incur a large overhead, so sampling is often employed. Samplers decide whether or not given span will be sampled. The default sampler is a probabilistic one with a configurable probability threshold. Users can write their own samplers as well.

CustomSampler sampler;

auto span = Span::StartSpan("Span", nullptr, {&sampler});



Trace Params
Trace params defines the tracing parameters used by opencensus. It sets the maximum number of attributes, annotations, message events, and links per span as well as the default probability sampler. Opencensus uses global trace params which can be set by the user. The values in the global trace params will be the default values for all spans unless specified otherwise.



Exporters
OpenCensus supports exporting of span data to various monitoring systems for analysis of tracing and stats data. The default exporter is stackdriver which is natively supported on Google Cloud Platform. OpenCensus also supports prometheus for stats and zipkin for tracing, and it allows for user supplied exporters.



Plugins
Opencensus supports integration with underlying rpc systems. By default opencensus works with grpc. Users can write their own plugins to interface opencensus with other rpc systems.



Views
A view is a data structure for handling on-task data collection. Once a View is instantiated, OpenCensus will collect data for it. Views must be registered for export to allow the data to be exported to backed monitoring systems like Stackdriver.



Measures
A measure defines one type of measurement, such as “RPC latency” or “HTTP request size”. Each view must have a measure which defines what is being measured and the units it is measured with.



Quickstart Example

The example demonstrates how to record stats and traces for a video processing system. It records data with the “frontend” tag so that collected data can be broken by the frontend user who initiated the video processing.

In this case we are using stdout exporters which we register at the beginning.

// Register stdout exporters.
opencensus::exporters::stats::StdoutExporter::Register();
opencensus::exporters::trace::StdoutExporter::Register();


We define a measure for video size which records the sizes in megabytes "MBy".
// Call measure so that it is initialized.
VideoSizeMeasure();


We create a view and register it with the local Stdout exporter.
// Create view to see the processed video size distribution broken down
// by frontend. The view has bucket boundaries (0, 256, 65536)
//that will group measure values into histogram buckets.
constopencensus::stats::ViewDescriptor video_size_view =
opencensus::stats::ViewDescriptor()
   .set_name(kVideoSizeViewName)
   .set_description("processed video size over time")
   .set_measure(kVideoSizeMeasureName)
   .set_aggregation(opencensus::stats::Aggregation::Distribution
   (opencensus::stats::BucketBoundaries::Exponential(8, 1 << 8, 2)))
   .add_column(kFrontendKey);

// Create the view.
opencensus::stats::View view(video_size_view);

// Register the view for export.
video_size_view.RegisterForExport();

In this case the view stores a distribution. The example records 1 video size to the view, which is 25648. This shows up in the histogram, with 1 bucket having a single value in it.

opencensus::stats::Record({{VideoSizeMeasure(), 25648}},{{kFrontendKey, "video size"}});

In this case there is no parent (root span), so the parent id is 0. There were 2 attributes added. After work has been completed a span must be ended by the user.
